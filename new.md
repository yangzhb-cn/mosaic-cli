## 1. 开发模型概述

软件开发流程通常遵循某种开发模型，常见的开发模型有三种：

### 瀑布模型（Waterfall）

```
需求分析 → 系统设计 → 编码实现 → 测试 → 部署 → 维护
```

- **特点**：阶段严格顺序执行，每个阶段有明确产出物
- **适用场景**：需求明确、变更少、复杂度低的小型项目
- **优点**：结构清晰、管理简单、文档完整
- **缺点**：变更成本高、用户反馈滞后

### 敏捷开发（Agile/Scrum）

```
每个迭代（1-4周）：
计划 → 设计 → 编码 → 测试 → 评审 → 回顾
```

- **特点**：短迭代、快速反馈、持续交付价值
- **适用场景**：需求变化快、需要快速上线的互联网产品
- **优点**：响应变化快、用户参与度高
- **缺点**：文档相对较少、对团队要求高

### DevOps 模型

```
计划 → 编码 → 构建 → 测试 → 发布 → 部署 → 运营
```

- **特点**：开发和运维一体化，强调自动化和持续交付
- **适用场景**：需要高频发布、对稳定性要求高的云原生项目
- **优点**：交付速度快、质量高、自动化程度高
- **缺点**：需要文化转变、工具链复杂

## 2. 阶段一：产品设计（需求分析）

### 2.1 核心目标

明确"要做什么"，回答产品要解决什么用户问题、创造什么价值。这是整个开发流程的起点，决定了项目成败。

### 2.2 主要活动

| 活动 | 内容 | 产出物 |
|------|------|--------|
| 市场调研 | 分析行业趋势、市场规模、竞品分析 | 市场调研报告 |
| 用户研究 | 用户访谈、问卷调研、用户画像 | 用户画像、用户旅程地图 |
| 需求收集 | 与客户/利益相关者沟通，明确业务目标 | 需求清单 |
| 需求分析 | 区分核心功能与期望功能，评估可行性 | 功能优先级矩阵 |
| 可行性分析 | 评估技术、经济、时间可行性 | 可行性报告 |

### 2.3 关键产出物

1. **BRD（商业需求文档）**：产品商业价值和 ROI 分析
2. **MRD（市场需求文档）**：目标市场、竞品分析、用户需求
3. **PRD（产品需求文档）**：详细功能定义、业务流程、逻辑规则
4. **原型设计**：
   - 线框图（低保真）：快速验证信息架构和布局
   - 高保真原型：模拟真实交互体验，用于用户测试
5. **用户故事地图**：从用户视角描述功能场景

### 2.4 常用工具

- **原型设计**：Figma、Sketch、Axure RP、墨刀
- **需求管理**：Notion、Confluence
- **思维导图**：XMind、MindNode
- **用户调研**：问卷星、腾讯问卷

### 2.5 最佳实践

- 尽量用可交互原型验证需求，不要等到开发后再改
- 需求必须有明确的验收标准（Acceptance Criteria）
- 建立需求变更管理流程，避免"范围蔓延"

## 3. 阶段二：系统设计

### 3.1 核心目标

明确"怎么做"，制定技术方案，为开发提供蓝图。

### 3.2 主要活动

| 活动 | 内容 | 产出物 |
|------|------|--------|
| 架构设计 | 选择技术栈，确定系统架构（单体/微服务/SOA） | 架构设计文档 |
| 模块划分 | 将系统分解为高内聚、低耦合的模块 | 模块结构图 |
| 数据库设计 | ER 图设计，表结构定义，索引策略 | 数据库设计文档 |
| 接口设计 | API 规范（REST/GraphQL/gRPC），数据格式定义 | API 文档（Swagger/OpenAPI） |
| UI/UX 设计 | 视觉设计、设计规范、组件库 | 设计稿、设计系统 |
| 安全设计 | 认证授权（OAuth/JWT）、数据加密、权限模型 | 安全设计文档 |

### 3.3 常见架构模式

- **单体架构**：适合小团队、简单应用
- **微服务架构**：适合大型系统、独立部署、技术栈灵活
- **分层架构**：表现层 → 业务层 → 持久层
- **事件驱动架构**：解耦服务，适合异步处理
- **前后端分离**：前端 SPA/SSR + 后端 API

### 3.4 设计评审要点

- 架构是否满足非功能性需求（性能、可用性、可扩展性）
- 数据库设计是否合理（范式化 vs 反范式化）
- API 是否遵循 RESTful 规范或 GraphQL 最佳实践
- 是否考虑异常场景和边界情况

### 3.5 常用工具

- **架构图**：Draw.io、Lucidchart、PlantUML
- **API 文档**：Swagger/OpenAPI、Postman
- **数据库设计**：MySQL Workbench、dbdiagram.io
- **设计协作**：Figma、Zeplin、Avocode

## 4. 阶段三：前端开发

### 4.1 核心目标

实现用户界面和交互逻辑，将设计稿转化为可交互的网页/应用。

### 4.2 技术栈选型

| 方向 | 技术选择 |
|------|----------|
| 框架 | React、Vue、Angular、Svelte |
| 状态管理 | Redux、Pinia、Zustand、Jotai |
| 样式方案 | Tailwind CSS、CSS Modules、Styled Components |
| 构建工具 | Vite、Webpack、Rollup、Parcel |
| 语言 | TypeScript（推荐）、JavaScript |
| 跨平台 | React Native、Flutter、Taro、uni-app |

### 4.3 前端工程化核心

1. **模块化开发**：ES Module 组织代码，提高复用性和可维护性
2. **组件化**：构建可复用的 UI 组件库，保持一致性
3. **构建打包**：自动化构建、代码压缩、Tree Shaking、代码分割
4. **代码规范**：
   - ESLint：代码质量检查
   - Prettier：代码格式化
   - Husky + lint-staged：提交前自动检查
5. **自动化测试**：
   - 单元测试：Jest、Vitest
   - 组件测试：Testing Library
   - E2E 测试：Playwright、Cypress
6. **CI/CD**：自动化构建、测试、部署到 CDN/服务器

### 4.4 性能优化

- 懒加载（Lazy Loading）和代码分割（Code Splitting）
- 图片优化（WebP/AVIF、懒加载、CDN）
- 缓存策略（Service Worker、HTTP 缓存）
- 首屏加载优化（SSR/SSG、预渲染）
- 虚拟列表处理大数据量渲染

### 4.5 最佳实践

- 使用 TypeScript 提高代码健壮性
- 语义化 HTML，关注无障碍（a11y）
- 响应式设计，适配多端
- 性能监控（Lighthouse、Web Vitals）

## 5. 阶段四：后端开发

### 5.1 核心目标

实现业务逻辑、数据处理、API 接口和系统集成。

### 5.2 技术栈选型

| 方向 | 技术选择 |
|------|----------|
| 编程语言 | Java（Spring Boot）、Go、Node.js、Python、Rust |
| 框架 | Spring Boot、Django、Express、FastAPI、Gin |
| 数据库 | MySQL/PostgreSQL（关系型）、MongoDB（文档）、Redis（缓存） |
| 消息队列 | RabbitMQ、Kafka、RocketMQ |
| 微服务 | Spring Cloud、gRPC、Service Mesh（Istio） |
| 容器化 | Docker、Kubernetes |

### 5.3 后端开发要点

1. **API 开发**：RESTful / GraphQL / gRPC 接口实现
2. **业务逻辑**：领域模型、服务层、事务管理
3. **数据访问**：ORM（MyBatis、Prisma、TypeORM）/ 原生 SQL
4. **认证授权**：JWT、OAuth 2.0、RBAC 权限模型
5. **安全防护**：SQL 注入防御、XSS/CSRF 防护、 Rate Limiting
6. **日志记录**：结构化日志（JSON）、日志级别管理
7. **错误处理**：统一异常处理、友好的错误码设计

### 5.4 代码质量保障

- **Code Review**：所有代码通过 Pull Request 审查
- **单元测试**：核心逻辑覆盖率高
- **静态分析**：SonarQube、Checkstyle
- **接口测试**：Postman/Newman 自动化测试

### 5.5 最佳实践

- 遵循 SOLID 原则和设计模式
- 编写清晰的 API 文档（Swagger/OpenAPI）
- 数据库连接池和慢查询优化
- 缓存策略（Redis 多级缓存）
- 熔断降级（Sentinel、Hystrix）

## 6. 阶段五：测试

### 6.1 核心目标

验证软件是否满足需求，发现并修复缺陷，确保质量。

### 6.2 测试金字塔

```
       ╱╲
      ╱ E2E ╲
     ╱────────╲
    ╱ 集成测试  ╲
   ╱──────────────╲
  ╱   单元测试     ╲
 ╱──────────────────╲
```

- **底层**：大量单元测试，快速验证单个函数/组件
- **中层**：集成测试，验证模块间协作
- **顶层**：少量 E2E 测试，验证完整业务流程

### 6.3 测试类型

| 测试类型 | 说明 | 工具 |
|----------|------|------|
| 单元测试 | 测试最小的代码单元（函数、方法） | JUnit、Jest、pytest |
| 集成测试 | 测试模块之间的交互 | Spring Boot Test、Supertest |
| 系统测试 | 测试整个系统功能 | 手动 + 自动化 |
| 性能测试 | 评估系统负载能力和响应时间 | JMeter、Locust、k6 |
| 安全测试 | 发现安全漏洞 | OWASP ZAP、Burp Suite |
| UI 测试 | 验证界面交互是否正确 | Selenium、Playwright |
| 用户验收测试（UAT） | 客户验证是否满足需求 | — |

### 6.4 测试流程

1. **测试计划**：确定范围、策略、资源、时间表
2. **测试用例设计**：基于需求和设计编写测试用例
3. **测试执行**：按计划执行测试，记录结果
4. **缺陷管理**：提交 Bug → 开发修复 → 验证关闭
5. **测试报告**：总结测试结果、质量评估

### 6.5 自动化测试策略

- 每次代码提交自动运行单元测试
- PR 时自动运行集成测试
- 部署前运行冒烟测试
- 定期运行完整回归测试

### 6.6 常用工具

- **测试管理**：Jira、TestRail、Xray
- **自动化**：Selenium、Playwright、Cypress
- **性能**：JMeter、k6、Locust
- **安全**：OWASP ZAP、Snyk、Dependabot
- **覆盖率**：JaCoCo、Istanbul

## 7. 阶段六：部署上线

### 7.1 核心目标

将经过测试的代码发布到生产环境，让用户可用。

### 7.2 环境管理

```
开发环境 (Dev) → 测试环境 (Test) → 预发布环境 (Staging) → 生产环境 (Production)
```

- **开发环境**：开发者本地
- **测试环境**：测试团队使用，数据隔离
- **预发布环境**：和生产环境一致，做最终验证
- **生产环境**：对用户提供服务

### 7.3 CI/CD 流水线

```
代码提交 → 自动构建 → 自动测试 → 自动部署到预发布 → 审批 → 部署到生产
```

**CI（持续集成）**：
- 每次提交自动构建和运行测试
- 快速发现集成问题
- 工具：Jenkins、GitHub Actions、GitLab CI

**CD（持续部署/交付）**：
- 自动部署到目标环境
- 部署策略控制风险
- 工具：ArgoCD、Spinnaker、Harness

### 7.4 部署策略

| 策略 | 说明 | 优点 | 适用场景 |
|------|------|------|----------|
| **蓝绿部署** | 维护两套环境，切换流量 | 回滚快、零停机 | 关键业务系统 |
| **金丝雀发布** | 先让小部分用户使用新版 | 风险可控、可观察 | 高风险变更 |
| **滚动更新** | 逐个/逐批替换实例 | 资源利用率高 | 微服务场景 |
| **功能开关** | 通过配置动态开关功能 | 精确控制、快速回滚 | 功能灰度 |

### 7.5 上线检查清单

- [ ] 数据库迁移脚本已验证
- [ ] 配置项已更新（环境变量、密钥）
- [ ] 依赖服务已就绪
- [ ] 监控和告警已配置
- [ ] 回滚方案已准备
- [ ] 通知相关方（客服、运营）
- [ ] 运行冒烟测试确认基本功能正常

### 7.6 常用工具

- **CI/CD**：Jenkins、GitHub Actions、GitLab CI、云效
- **容器化**：Docker、Kubernetes（K8s）
- **IaC**：Terraform、Ansible、Pulumi
- **部署**：ArgoCD、Helm、Kustomize
- **制品管理**：Docker Hub、Harbor、Nexus

## 8. 阶段七：运维维护

### 8.1 核心目标

保障系统稳定运行，持续优化和改进。

### 8.2 可观测性三大支柱

| 维度 | 说明 | 工具 |
|------|------|------|
| **指标（Metrics）** | CPU、内存、QPS、延迟等数值 | Prometheus、Grafana |
| **日志（Logs）** | 应用和系统运行日志 | ELK Stack（Elasticsearch+Logstash+Kibana）、Loki |
| **链路追踪（Traces）** | 请求在各服务间的完整路径 | Jaeger、Zipkin、SkyWalking |

### 8.3 运维主要工作

1. **监控告警**
   - 基础设施监控：服务器、网络、数据库
   - 应用监控：APM（New Relic、Datadog、SkyWalking）
   - 业务监控：核心业务指标（订单量、转化率）
   - 告警规则：设置合理的阈值，避免告警风暴

2. **日志管理**
   - 集中式日志收集
   - 结构化日志（JSON 格式）
   - 日志分级（Debug/Info/Warn/Error）
   - 日志留存策略

3. **故障处理**
   - 故障发现 → 定位 → 恢复 → 复盘
   - 建立 On-Call 值班机制
   - 事后撰写故障报告（Postmortem）

4. **备份与容灾**
   - 数据库定期备份
   - 跨区域容灾
   - 自动故障转移

5. **容量规划**
   - 监控资源使用趋势
   - 预测未来容量需求
   - 自动扩缩容（HPA）

6. **安全管理**
   - 依赖漏洞扫描（Snyk、Dependabot）
   - 定期安全审计
   - 密钥管理（Vault、AWS Secrets Manager）

7. **基础设施即代码（IaC）**
   - 使用 Terraform / Pulumi 管理云资源
   - 使用 Ansible / Chef 配置管理
   - 确保环境一致性和可复现性

### 8.4 SRE 核心原则

- **服务等级目标（SLO）**：定义可接受的服务可用性
- **错误预算（Error Budget）**：在可靠性和发布速度之间做权衡
- **减少重复劳动**：自动化一切可自动化的操作
- **事后复盘**：不追究责任，重在改进流程

### 8.5 常用的运维工具

| 类别 | 工具 |
|------|------|
| 监控 | Prometheus、Grafana、Zabbix、Nightingale |
| 日志 | ELK Stack、Loki、Splunk |
| APM | SkyWalking、Datadog、New Relic |
| 告警 | AlertManager、PagerDuty |
| IaC | Terraform、Ansible、Pulumi、Crossplane |
| 容器编排 | Kubernetes、Docker Compose、Nomad |
| 密钥管理 | HashiCorp Vault、AWS Secrets Manager、1Password |

## 11. 总结

### 软件开发全流程闭环

```
┌──────────────────────────────────────────────────────┐
│                    用户反馈 / 数据驱动                  │
│          ↓                                             │
│  需求分析 → 系统设计 → 开发 → 测试 → 部署 → 运维      │
│          ↓                                             │
│                  持续迭代                               │
└──────────────────────────────────────────────────────┘
```

### 关键成功因素

1. **清晰的需求定义**：避免范围蔓延，确保团队目标一致
2. **有效的沟通机制**：站会、评审、回顾持续对齐
3. **自动化优先**：自动化测试、构建、部署减少人为错误
4. **质量内建**：从需求阶段就考虑质量，而不是最后才测试
5. **安全左移**：在开发早期就考虑安全
6. **用户中心**：持续收集反馈，数据驱动决策
7. **持续改进**：每个迭代结束复盘，不断优化流程

### 学习路径建议

1. **入门**：选择一个小项目，完整走一遍"需求 → 设计 → 开发 → 测试 → 部署"流程
2. **深入**：学习敏捷/Scrum 方法论，实践 Code Review 和 CI/CD
3. **进阶**：研究微服务架构、容器化、Kubernetes 编排
4. **专家**：掌握 SRE 实践、可观测性体系、大规模系统设计
